Semantic Markup

Why use Semantic Markup? #1

Consider "what purpose or role does that HTML element have?
Example Medium.com
Once inspected, there's many mark-ups with divs - these are meaningless mark-ups as containers that don't do anything
A better example is - Stripe.com
There is an element called 'main' which is the main content of the page - where Medium site would just have had 'divs'
Then it has a header element - functions just like a div but has a definition
Then there's a 'section' element that works like a div but has a meaning too
Then 'nav' elements which again do what a div but tell us (meaningful) that they are used for navigation
These better defined elements are newer e.g. another one is a 'footer' element
So on medium, it works, but you can't tell when developing what the content is. Easy to get lost in the code!
Why does it matter? Helps for SEO (Search Engine Optimisation) - so e.g. a 'crawler' can look at all the links in a nav page
So if you just used 'divs' it's not as easy to tell where the links for navigation are

A lot of these elements don't do anything different to divs - they just make it clearer what they are!

Why use Semantic Markup? #2

Using semantic elements also makes code more accessible - e.g. screen readers can use these as hooks /signposts
So it can differentiate between e.g. you want to go to the about page, the careers page, you can ask for certain semantic elements
e.g. going to the footer or to the navigation bar to get that info

Why use Semantic Markup? #3

It makes your code easier to read yourself and organised!
If you want to update a certain section you know where to go easier

Some examples of semantic elements (to use instead of divs / spans) are:

  • Section
    • Represents a standalone section - which doesn't have more specific semantic element to represent it
      Typically sections have a heading
  • Article
    • Represents a self-contained composition in a document/page intended to be individually distributable or reusable
      e.g. a widget for weather forecast (see below)
      Examples include: a forum post, a magazine or newspaper article, a blog entry
      There should be a header that says what the article is about
  • Nav
    • Mozilla on Nav
    • The HTML nav element represents a section of a page whose purpose is to provide navigation links,
      either within the current document or to other documents.
      Common examples of navigation sections are menus, tables of contents, and indexes.
  • Main
    • The main element - represents the dominant body in the document - should exclude things that would be repeated across pages
      For example, not sidebars, navigation bars, headers, footers etc.
  • Header
    • Any sort of introductory content - can also include navigation content (including Nav elements as children?)
      Note you can also put more than one Headers and Footers within other elements e.g. within an Article
  • Footer
    • Ending content - will also often include 'Nav' elements as children
  • Aside
    • An aside is a portion of a document whose content is only indirectly related to the document's main context
      i.e. a sidebar or call-out box
  • Summary
    • x
  • Details
    • x
  • Time
    • This is a bit different - an inline element that wraps around content that is date or time
      Just helps this datetime attribute to make machine readable
  • Figure
    • Usually contains an image - and possibly a captain

A good resource is: Mozilla on semantics
