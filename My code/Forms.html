The form element - creating forms

The form element itself is a shell or container that doesn't have any visual impact
E.g. inputs for a login screen
E.g. Inputs for name, surname, email, password inputs
E.g. radio button to select a subscription type
e.g. a form for a search bar within a webpage! 'search this page'
The form element is an input to group together - and it dictates where the form data should be sent when it's submitted!
So from that perspective it's a bit more useful than some of those other elements

  • The 'action attribute' within the form element specifies where the form data should be sent
    • For example if you set it as action = "/search/" then it will submit the form to
      e.g. www.websitename.com/search/?q=inputoftheform
      i.e. relative to the website we're currently on
      Note that spaces are adjusted to '%20' in the url
    • When you submit a form, a HTTP request is made. The method determines the type
  • The 'method' attribute specifies which HTTP method should be used (to be covered later)
    • 'Get' requests and
    • 'Post' requests - forget this for now will be covered later!

Form validations - constraints

e.g. your password must have x y z (special character, a number, etc)
but this is the built in validations for forms - others would need to be using javascript (smoother experiences)
e.g. on Stripe Registration site the 'excellent password' popup

This is all 'client side validation' - which is done within the browser
There is also something called 'server side validation' once a form is submitted
To make sure the data meets the server's expectations. e.g. if someone used a terminal to get around client side validation.
This was historically the way it was done - input details, press submit, then server would tell you if any issues.
Client side validation adds to UX but both are needed really.
